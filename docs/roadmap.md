# GoodGame 跑团 Web Demo 迭代规划

## 项目目标

- 这是一个纯前端的跑团 Demo，不包含后端。
- AI 负责 DM 叙事与选项生成，玩家只通过点击选项推进剧情。
- 掷骰随机数必须由程序生成，不能由大模型生成。
- 对话输入由多块上下文拼装而成：世界观、角色状态、NPC 状态、参考案例、近期回合。

## 已确认技术方向

- **LLM 接入方式**：纯前端直连第三方 API（Demo 阶段，不考虑 API Key 安全）。
- **接口兼容**：同时支持 OpenAI `Completions`（旧）和 `Responses`（新）两种调用格式。
- **模式切换**：在设置页中可切换 `completion` / `responses`。
- **输出格式**：强制 JSON；不合规则不推进回合。
- **状态变更**：AI 只能给出变更提案，必须玩家手动确认后才写入状态。
- **页面布局**：游玩页采用左叙事、右状态面板的双栏布局。
- **数据存储**：纯前端本地存储，优先使用 IndexedDB，并支持导入/导出。

## 版本拆分

## v0.1（首个可玩版本）

### 功能范围

1. 上下文系统（最小可用）
- 基础世界观文本管理。
- 玩家角色（属性/技能/状态/装备/道具）管理。
- NPC（状态/关系/历史摘要/属性等）管理。
- 参考案例库（手动启用/禁用与排序）。

2. 核心游玩循环
- AI 产出叙事与多个可选项。
- 玩家点击选项推进。
- 若选项带判定，则触发程序掷骰（正则解析通用骰表达式，如 `1d20+3`）。
- 把玩家选择与掷骰结果写入回合记录，再继续请求 AI。

3. 状态变更提案流
- AI 返回 `proposedPatches`。
- 右侧状态面板展示变更提案（可确认/拒绝）。
- 仅确认后的提案才写入角色/NPC/世界状态。

4. API 适配
- `POST /v1/completions` 适配层。
- `POST /v1/responses` 适配层。
- 统一成内部响应结构供 UI 使用。

5. 本地数据能力
- 使用 IndexedDB 存储配置、设定、会话、回合历史。
- 提供全量导入/导出 JSON。

### UI 范围

- `/play`：双栏布局（左叙事与选项，右状态与提案）。
- `/context`：世界观与上下文拼装策略。
- `/pc`：玩家角色编辑。
- `/npcs`：NPC 管理。
- `/cases`：参考案例库。
- `/settings`：API 连接与接口模式（completion/responses）配置。

### v0.1 验收标准

- 能从空配置启动并完成至少 20 回合连续游玩。
- 每次掷骰有程序日志（骰表达式、每颗骰子结果、总值、成功/失败）。
- 模型返回不合规 JSON 时能阻断并提示重试。
- 状态变更不会自动生效，必须经玩家确认。
- 刷新页面后会话可恢复，导出后可在同浏览器重新导入继续。

## v0.2（上下文增强）

- 近期回合摘要、长短期记忆分层。
- token 预算估算与超限策略（截断/降级）。
- 案例库增强筛选（标签与手动优先级优化）。

## v0.3（规则与判定深化）

- 通用判定模板扩展（对抗检定、成功等级细分）。
- 更细粒度状态系统（效果持续回合、条件触发）。
- 规则集预设入口（为未来 DND/COC 等模板留位）。

## 开发流程约定

1. 先完成 v0.1 并提交验收。
2. 你验收通过后再进入 v0.2。
3. 每次迭代先更新本路线图与对应协议文档。
